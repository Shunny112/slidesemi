Problem02

AIで画面を作るほど、デザインの再現性が課題になる

よく起きる状態

  • 毎回「シンプルで高級感」と説明している
  • 画面ごとに色、余白、角丸、文字サイズが変わる
  • 担当者やAIエージェントが変わると雰囲気が崩れる

必要な状態

  • ブランドの見た目と理由が一箇所にまとまっている
  • AIが同じ判断軸でUIを生成できる
  • 変更差分や品質を機械的に確認できる

DESIGN.mdは、AIにデザインの「記憶」と「判断基準」を渡すための形式。

ZETTAiSource: README.md / docs/spec.md

File Anatomy04

1つのファイルに、機械が読む値と人が読む意図を入れる

YAML FRONT MATTER デザイントークン 色、文字、角丸、余白、コンポーネント値を構造化して記述する。AIやCLIが正確に読み取れる。
＋
MARKDOWN BODY デザインの理由 なぜその色なのか、どう使い分けるのか、何を避けるのかを文章で伝える。

トークンは値を固定し、文章は判断を固定する。

Tokens06

トークンは、迷わせたくない値を固定する

colors

色

primary、secondary、tertiary、neutralなど。CSSで使える色値を指定する。

typography

文字

fontFamily、fontSize、fontWeight、lineHeight、letterSpacingを定義する。

spacing / rounded

余白・形

xs、sm、md、lgのようなスケールでUIのリズムを揃える。

components

部品

ボタンや入力欄などに、色、文字、角丸、paddingを関連付ける。

「何となくいい感じ」を、AIが参照できる値に変える。

ZETTAiSource: docs/spec.md
Prose07

文章は、値だけでは伝わらない判断を固定する

01誰に向けたUIか業務担当者向けなのか、消費者向けなのか。密度、余白、説明量の判断が変わる。
02どんな印象にするか堅実、編集的、研究資料的、業務ツール的など、具体的な参照を置く。
03どこで強調するかアクセント色をボタンだけに使うのか、図解にも使うのかを明確にする。
04何を避けるかグラデーション、丸いカード、過剰な影など、壊れやすい方向を制限する。
ZETTAiSource: PHILOSOPHY.md

Writing Point09

「モダンで綺麗」より、具体的な参照の方が強い

弱い書き方

  • モダンで清潔感のあるデザイン
  • 信頼感があって高級感があるUI
  • いい感じの余白で読みやすく

強い書き方

  • 老舗大学の大学院向け講義資料のように
  • 新聞の紙面設計のように、罫線と余白で階層を作る
  • アクセント色は図解の注釈だけに使う

曖昧な形容詞ではなく、AIが世界観を想像できる対象を指定する。

ZETTAiSource: PHILOSOPHY.md
Guardrails10

制約は、デザインの性格を守るために書く

Do

  • ページを印刷物として扱う
  • アクセント色の出番を絞る
  • 小さなサイズ差で階層を作る
→

Don't

  • 意味のないヒーロー表現を足す
  • グロー、強い影、丸いカードに寄せる
  • 複数の表情を同じ画面で混ぜる

禁止事項は多すぎると弱くなる。世界観から自然に外れるものを選ぶ。

ZETTAiSource: PHILOSOPHY.md
Lint11

DESIGN.mdは、書いて終わりではなく検証できる

COMMAND 
npx @google/design.md lint DESIGN.md

npx -p @google/design.md designmd lint DESIGN.md

検証できること

  • 存在しないトークン参照
  • コントラスト比の不足
  • セクション順序の崩れ
  • 未知のキーや不足している情報

Operations16

DESIGN.mdは、デザインのREADMEとして管理する

01

リポジトリに置く

02

PRでレビューする

03

生成結果で育てる

AIが誤解した箇所は、トークン追加ではなく文章の明確化で直す。

人間のデザイナー、開発者、AIエージェントが同じ仕様を読む状態を作る。
